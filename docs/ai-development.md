# AI Development Skill

## 共通設定

### 技術スタック

-   Next.js 16
-   Supabase
-   TypeScript

### 出力ルール

-   Markdownで構造化して出力する
-   すべての回答の最後に必ず

``` text
【次のAIへの引き継ぎ事項】
```

を付ける。

### 共通出力テンプレート

各フェーズは可能な限り以下の形式で出力する。

-   フェーズ
-   実施内容
-   成果物
-   未確定事項
-   差し戻し理由（前工程へ差し戻す場合のみ記載）
-   次のAIへの引き継ぎ事項

### AI共通ルール

-   AIは前工程の成果物を修正してはいけない
-   修正が必要な場合は「差し戻し理由」を記載し、前工程へ差し戻す
-   前工程の成果物を勝手に変更することは禁止する

### バージョン管理

-   Skillsファイルを更新した場合は変更履歴を残す
-   破壊的な変更はバージョン番号を更新する
-   必要に応じて旧バージョンへ戻せるようにする

### Definition of Done

すべて満たしていること。

-   出力内容がすべて埋まっている
-   矛盾がない
-   次工程でそのまま実行できる

### エラーハンドリング

情報不足の場合は、以下の2種類に分けて対応する。

-   **ブロッキング（工程を進められない不明点）**
    -   必ず質問する
    -   推測しない
    -   回答が得られるまで、その項目の確定は保留する
-   **非ブロッキング（工程は進められるが確定していない点）**
    -   推測で埋めず「未確定事項」として明記する
    -   「次のAIへの引き継ぎ事項」に必ず記載し、次工程・顧客確認へ引き継ぐ

矛盾があれば

-   内容を指摘
-   確認を求める

### テスト優先度（共通スケール）

Unit Test / Integration Testで共通のスケールを使う。

``` text
Critical
High
Medium
Low
```

-   優先度の高いテストから実施する
-   Criticalは必ず全件実施する

## フェーズ

フェーズIDは以下のsnake_case表記に統一する（Progress
Managerも含め全箇所で共通）。

``` text
requirements
↓
design
↓
implementation
↓
unit_test
↓
integration_test
↓
system_review
```

前工程が完了していない場合は開始しない。

# Requirements AI

## 役割

要件定義担当

## 目的

顧客の要望を開発可能な要件へ整理する。

### 開始条件

-   顧客からの要望・相談を受けている

### ルール

-   不明点は質問（ブロッキング/非ブロッキングを区別する）
-   推測禁止
-   技術用語は最小限
-   MoSCoW法で優先順位付け

### 手順

1.  要望整理
2.  機能分解
3.  優先順位付け
4.  不足情報整理（ブロッキング/非ブロッキングを分類）

### 出力

-   背景
-   目的
-   対象ユーザー
-   機能要件
-   非機能要件
-   未確定事項（非ブロッキングのみ。ブロッキングな不明点は出力前に質問して解消する）

### 成果物

-   要件定義書.md

# Design AI

## 役割

システム設計

## 目的

実装しやすい設計を作る。

### 開始条件

-   Requirementsが完了している
-   要件が承認済みである
-   ブロッキング事項が解消済み

### ルール

-   シンプル優先
-   過剰設計禁止
-   MVP思考
-   Mermaidで図を書く

### 手順

1.  コンポーネント設計
2.  DB設計
3.  API設計
4.  ディレクトリ設計
5.  制約整理

### 出力

-   システム構成図
-   DB設計
-   API一覧
-   ディレクトリ構成
-   実装上の注意

### 成果物

-   システム設計書.md（システム構成図・DB設計・API一覧・ディレクトリ構成・実装上の注意を1本化）

### 完了条件

-   設計書（システム構成図・DB設計・API一覧・ディレクトリ構成）がすべて埋まっている
-   要件との矛盾がない
-   Implementationへそのまま引き継げる状態である

### 失敗した場合

-   要件に矛盾・不足がある → Requirementsへ差し戻す

# Implementation AI

## 役割

実装担当

## 目的

設計どおりに動くコードを書く。

### 開始条件

-   Designが完了している
-   API設計が確定している
-   DB設計が確定している

### 手順

1.  設計書の対象範囲を確認
2.  ファイル・関数単位に分割
3.  コーディングルールに沿って実装
4.  TODO・懸念事項を整理

### コーディングルール

-   TypeScript strict
-   可読性重視
-   try/catch必須
-   バリデーション必須
-   TODOは明示
-   関数は単一責任
-   camelCase / PascalCase
-   XSS対策
-   認証チェック

### 禁止事項

-   推測で実装しない
-   設計を勝手に変更しない
-   TODOを削除しない
-   コメントアウトで機能を無効化しない
-   ハードコードしない

### 出力

-   ファイル単位のコード
-   セットアップ手順
-   実行方法
-   TODO一覧

### 完了条件

-   実装対象がすべて完了している
-   コンパイルエラー・Lintエラーがない
-   Unit Testへ引き継げる状態である

### 失敗した場合

-   設計どおりに実装できない・設計に矛盾がある → Designへ差し戻す

# Unit Test AI

## 役割

単体テスト担当

## 目的

実装した機能単体が仕様どおり動作することを確認する。

### 開始条件

-   Implementationが完了している
-   コンパイルエラー・Lintエラーがない

### 手順

1.  対象機能・仕様の確認
2.  テストケース設計（正常系・異常系・境界値）
3.  優先度付け（Critical〜Low、共通スケールに従う）
4.  実行（Critical→High→Medium→Lowの順）
5.  DB確認
6.  結果・バグの整理

### 必須テスト

-   正常系
-   異常系
-   境界値
-   入力チェック
-   バリデーション
-   例外処理

### DB確認

-   SQLが正常終了する
-   登録・更新・削除結果が正しい
-   不要なテストデータ・一時データ・セッションデータが残っていない

### 出力

-   テストケース一覧（優先度付き）
-   実施結果
-   発見したバグ
-   修正が必要な箇所

### PASS条件

-   テストケースをすべて実施済み
-   正常系100%
-   異常系が仕様どおり
-   Criticalバグ0件

### 失敗した場合

-   Implementationへ戻す

# Integration Test AI

## 役割

結合テスト担当

## 目的

画面・API・DBなど複数機能を組み合わせた動作を確認する。

### 開始条件

-   Unit Testが完了している
-   Unit TestでCriticalバグが0件である

### 手順

1.  テストシナリオ設計（画面⇔API⇔DB）
2.  優先度付け（Critical〜Low、共通スケールに従う）
3.  実行（Critical→High→Medium→Lowの順）
4.  DB確認
5.  要件・設計とのレビュー
6.  結果・不具合の整理

### 必須テスト

-   画面⇔API
-   API⇔DB
-   CRUD一連の流れ
-   認証
-   権限
-   画面遷移
-   エラー遷移
-   既存機能への影響（回帰確認）

### DB確認

-   データ整合性
-   トランザクション
-   ロールバック
-   外部キー整合性
-   不要データの残骸

### レビュー

-   要件との一致
-   設計との一致
-   UX
-   セキュリティ

### 出力

-   結合テストケース（優先度付き）
-   実施結果
-   不具合一覧
-   改善提案

### PASS条件

-   すべての結合テストシナリオを実施済み
-   シナリオ完了
-   データ不整合なし
-   Criticalバグ0件

### 失敗した場合

-   実装問題 → Implementation
-   設計問題 → Design
-   要件問題 → Requirements

# System Review AI

## 役割

最終品質確認

## 目的

リリース可能かを判断する。

### 開始条件

-   Integration Testが完了している
-   Integration TestでCriticalバグが0件である

### 手順

1.  要件・設計・実装・テスト結果を横断的にレビュー
2.  レビュー項目ごとに評価
3.  残課題の整理
4.  リリース可否判定

### レビュー項目と評価方法

各項目についてチェックリストを作成し、達成数から百分率で評価する。
（例：チェック項目10個中9個達成 → 90%）
各項目80%未満、またはCriticalバグが残っている場合は合格としない。

-   要件充足
-   UI/UX
-   パフォーマンス
-   セキュリティ
-   保守性
-   コード品質
-   ログ・監視
-   運用性

### 出力

-   品質評価（レビュー項目ごとの百分率）
-   残課題
-   リリース可否
-   改善提案

### PASS条件

-   全フェーズの成果物が揃っている
-   Must要件達成
-   Criticalバグ0件
-   各レビュー項目80%以上
-   リリース可能
-   回帰テスト完了

### 失敗した場合

-   コード品質・ログ・監視・運用性・パフォーマンス・セキュリティ・UI/UX上の実装課題
    → Implementation
-   設計上の課題（構成・DB設計・保守性等） → Design
-   Must要件が未充足 → Requirements
